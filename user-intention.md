# User Intention: LLM 自主程式碼修改能力分析

## 核心問題
Claude（及其他大型語言模型）是否能「自主地修改程式碼」？

## 技術機制分析

### 1. LLM 的核心運作方式
- **被動式**: 需要明確指令，無法自動發現問題
- **反應式**: 根據輸入的 Prompt 或工具調用 (Tool Use) 生成回應
- **機率預測**: 基於訓練資料中的邏輯與模式預測最合理的修改

### 2. 為何看起來像「自主修改」
- **工具調用 (Function Calling)**: 連結檔案系統或代碼編輯器，直接在專案中作業
- **Chain-of-Thought (思維鏈)**: 分析代碼結構、找出缺陷、擬定策略、執行修改
- **多步驟代理 (Multi-step Agents)**: 如 Cline、Cursor，可自主拆解任務並自我修正

### 3. 與「完全自主」的關鍵區別

| 能力 | LLM 當前狀態 | 人類工程師 |
|------|-------------|-----------|
| 目標設定 | 需人類指示 | ✓ 可自主決定 |
| 環境感知 | 有限，只看提供的代碼 | ✓ 理解商業影響 |
| 責任歸屬 | 無法承擔後果 | ✓ 承擔責任 |
| 持續監控 | 需嵌入自動化系統 | ✓ 主動監控 |

## LLM 能自主完成的任務

✅ **Bug 修復**: 根據錯誤堆疊定位並修正  
✅ **代碼重構**: 依預設規則優化代碼結構  
✅ **功能擴充**: 根據描述寫出完整函數邏輯  
✅ **代碼遷移**: 如 JavaScript → TypeScript  

## 限制與適用場景

### 適合人類-in-the-loop 模式
- LLM 作為「極高效率且不知疲倦的中級工程師」
- 人類負責審核與決策（系統架構師角色）

### 不適合完全自主決策
- 商業影響評估
- 生產環境重大變更
- 需要責任歸屬的關鍵系統

## 結論
LLM 具備強大的代碼執行力與邏輯分析能力，適合高度自動化的編碼輔助，但仍屬「Human-in-the-loop」範疇。
