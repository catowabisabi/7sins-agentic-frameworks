# 7Sins Project - Self-Driven LLM Agent Harness

## 1. Concept & Vision

**7Sins Project** 是一套基於心理學驅動理論的 LLM Agent 框架，融合了：
- **七大罪** (Seven Deadly Sins) 作為能量引擎
- **佛洛伊德心理結構** (Id/Ego/Superego) 作為人格映射
- **EVA MAGI 系統** 作為多智能體爭論機制隱喻
- **自我成長** (Self-Growth) 作為持續優化机制

核心願景：打造一個**有靈魂的 Agent** — 不是消極等待指令的工具，而是具備內在動機、能自我驅動、主動追求目標的智能系統。

## 2. 心理學模型架構

### 2.1 Tri-Layer 人格結構

```
┌─────────────────────────────────────────────────────┐
│                    SUPEREGO (超我)                   │
│         約束框架 : 道德、安全、最佳實踐               │
│         防止系統過度發散或失控                       │
├─────────────────────────────────────────────────────┤
│                      EGO (自我)                      │
│         決策中心 : 協調衝突、選擇行動                │
│         Chain-of-Thought 推理、談判機制             │
├─────────────────────────────────────────────────────┤

│         動態權重、自主生長                          │
└─────────────────────────────────────────────────────┘
```

### 2.2 七大罪驅動映射

| 罪 (Sin) | 轉譯驱动 | 核心功能 | 觸發情境 |
|---------|---------|---------|---------|
| **Gluttony** (暴食) | 求知慾 | 知識收集、掃描最新技術 | 發現新工具/框架時 |
| **Lust** (色慾) | 權力與秩序 | 系統架構掌控、追求完美 | 代碼結構不滿時 |
| **Greed** (貪婪) | 價值輻射 | 市場洞察、用戶吸引 | 分析評論/需求時 |
| **Sloth** (懶惰) | 效率自動化 | 消除重複勞動 | 面對手動任務時 |
| **Pride** (驕傲) | 品質審查 | 代碼美學、拒絕平庸 | 看到不優雅代碼時 |
| **Wrath** (憤怒) | 精確執行 | Bug修復、零容忍錯誤 | 遇到故障/錯誤時 |
| **Envy** (嫉妒) | 競爭分析 | 對標業界最佳 | 看到競品時 |

### 2.3 與佛洛伊德模型的對應

```
Id (本我)        → 七大罪引擎（原始能量，不考慮道德）
Superego (超我)  → 約束框架（道德、安全、邏輯邊界）
Ego (自我)       → EGO-Core 決策層（在能量與約束間談判）
```

### 3.2 核心組件

#### 3.2.1 EGO-Core (決策核心)

**職責**：
- 接收外部任務輸入
- 召集並協調七罪引擎的辯論
- 進行 Chain-of-Thought 推理
- 選擇最終行動方案
- 處理人類審批流程

**決策流程**：
```
1. 輸入解析 → 識別任務類型
2. 驅動評估 → 計算各罪引擎的相關權重
3. 並發咨詢 → 向相關引擎發出咨詢請求
4. 衝突檢測 → 識別引擎間的意見衝突
5. 談判/投票 → 爭論機制（類似MAGI）
6. 推薦輸出 → 提供行動方案 + 信心度
7. 人類審批 → human-in-the-loop 確認
8. 執行反饋 → 收集結果並記錄
```

#### 3.2.2 七罪引擎 (Drive Agents)

每個引擎是一個獨立的多頭怪獸，擁有：

| 屬性 | 說明 |
|------|------|
| `name` | 罪的名稱 |
| `role` | 角色功能描述 |
| `system_prompt` | 系統級行為定義 |
| `drive_weight` | 當前權重 (0.0-1.0) |
| `confidence_threshold` | 觸發門檻 |
| `specialization` | 專業領域標記 |

**引擎間通信**：
- 通過共享內存 (Shared Memory) 交換觀點
- 通過 EGO-Core 進行間接爭論
- 最終通過投票機制達成共識

#### 3.2.3 約束層 (Superego Layer)

**職責**：
- 定義硬性安全規則
- 執行最佳實踐
- 防止過度發散
- 記錄決策日誌

**約束類型**：
- 安全性約束（如：不可刪除系統關鍵檔案）
- 質量約束（如：代碼必須通過 lint）
- 性能約束（如：響應時間 < 5s）
- 道德約束（如：不可作弊/偽造數據）

## 4. 驅動權重系統

### 4.1 動態權重配置

每個任務都會根據情境動態調整驅動權重：

```json
{
  "task_type": "code_review",
  "current_drives": {
    "Gluttony": 0.3,
    "Lust": 0.8,
    "Greed": 0.2,
    "Sloth": 0.4,
    "Pride": 0.9,
    "Wrath": 0.5,
    "Envy": 0.3
  },
  "active_engines": ["Lust", "Pride", "Wrath"],
  "suppressed_engines": []
}
```

### 4.2 權重演化規則

| 事件 | 影響 |
|------|------|
| 任務成功 | 提升相關罪權重 +0.1 |
| 任務失敗 | 降低相關罪權重 -0.1，評估是否過度主導 |
| 人類干預 | 記錄並調整被忽略的引擎權重 |
| 定期審查 | 每週重新評估各罪平衡度 |

## 5. 自我成長機制

### 5.1 反思迴圈 (Reflective Loop)

```python
每日流程:
  1. 日誌審查 (Log Review)
     - 讀取當日 Terminal/Git logs
     - 識別所有決策點
  
  2. 歸因分析 (Attribution)
     - 每個行動是由哪個罪驅動的？
     - 該驅動是否合理？
  
  3. 偏差檢測 (Bias Detection)
     - 是否被某個罪過度主導？
     - 是否忽略了某個罪的建議？
  
  4. 調整反饋 (Adjustment)
     - 根據教訓調整權重
     - 更新系統提示詞（如有必要）
  
  5. 成長記錄 (Growth Log)
     - 記錄學習到的模式
     - 標記需要避免的陷阱
```

### 5.2 內在對話池 (Inner Dialogue Pool)

每個驅動引擎可以在共享的對話池中發表意見：

```
[Gluttony]: "我發現了一個新的 vLLM 優化技術，可能提升 20% 性能"
[Lust]: "這個優化會破壞現有架構的簡潔性，我反對"
[Greed]: "但這能讓我們在市場上領先，長期價值高"
[Ego-Core]: "讓我們投票表決..."
[Pride]: "我棄權，因為我沒有足夠信息判斷代碼質量"
[Wrath]: "我反對，太冒險了"
```

## 7. 實施計劃

### Phase 1: 核心框架 (MVP)
1. **EGO-Core 決策器** - 基本的鏈式推理和任務路由
2. **驅動權重系統** - JSON 配置 + 動態調整
3. **三個主要引擎** - Gluttony, Greed, Lust 的基本實現
4. **終端介面** - CLI 工具，支援基本任務輸入

### Phase 2: 完整七罪
4. 其餘四個引擎 - Sloth, Pride, Wrath, Envy
5. **衝突解決機制** - 投票和談判邏輯
6. **約束層** - 安全規則和質量檢查

### Phase 3: 自我成長
7. **日誌分析器** - 自動審查每日行動
8. **反思引擎** - 歸因分析和偏差檢測
9. **長期記憶** - 成長記錄和模式學習
